# 발표 데모 체크리스트

## 목적

- 발표 중 데모 실패 가능성을 줄이고, 보여줄 흐름을 짧고 안정적으로 고정합니다.
- 제품 앱(`apps/web`)을 기준 데모로 사용하고, 필요 시 `docs/prototypes/`를 백업 시나리오로 사용합니다.

## 발표 직전 확인

1. 메인 브랜치가 최신인지 확인합니다.
2. 현재 로컬 실행 환경의 API 주소가 맞는지 확인합니다.
3. 발표용 자산 폴더가 준비됐는지 확인합니다.
   - `docs/presentation/assets/app`
   - `docs/presentation/assets/model`
4. 브라우저에서 기존 세션이 꼬였으면 로그아웃 후 다시 로그인합니다.
5. 발표용 계정 정보를 준비합니다.
   - 이메일: `demo-owner@example.com`
   - 비밀번호: `secret123!`
6. 네트워크 상태가 불안하면 `docs/prototypes/web-flow-prototype.html`도 바로 열 수 있게 준비합니다.

## 메인 데모 시나리오

### 1. 로그인

- `/login` 진입
- 데모 계정 채우기 또는 미리 준비한 계정으로 로그인
- 로그인 후 메인 작업 화면 진입 확인

### 2. 콘텐츠 생성

- 업종, 위치, 목적, 스타일, 채널을 선택합니다.
- 이미지 업로드를 진행합니다.
- 생성 버튼을 눌러 결과 화면까지 이동합니다.

### 3. 결과 확인

- 영상 결과
- 게시용 이미지
- 캡션 후보와 해시태그
- 빠른 수정 옵션
- 필요하면 `docs/presentation/assets/app/generated-video.mp4`를 백업 증거물로 사용

위 4가지를 짧게 보여줍니다.

### 4. 게시 fallback

- 자동 게시가 항상 보장되는 구조가 아니라는 점을 설명합니다.
- 필요하면 `assist_required` 흐름으로 전환되는 점을 보여줍니다.
- “자동 게시 실패 시 업로드 보조 방식으로 이어진다”는 점을 강조합니다.

### 5. 계정 연결 / 이력

- 계정 연결 상태 화면에서 채널 상태를 보여줍니다.
- 이력 화면에서 기존 작업을 다시 여는 흐름을 보여줍니다.

## 발표 중 강조 포인트

1. 팀원 작업을 폴더 단위로 합치지 않고 기능 단위로 선별 통합했습니다.
2. 로그인, 생성, 결과 확인, 게시 fallback, 계정 연결까지 하나의 trunk에서 이어집니다.
3. 테스트와 빌드 검증을 통과한 기준선입니다.

## 인정해야 할 한계

1. 일부 도메인 데이터는 아직 demo-state 중심입니다.
2. 모델 실험은 trunk 안에 스냅샷으로 보존했지만, 실제 고성능 추론은 별도 GPU 환경이 필요합니다.
3. 현재 앱에서 돌아가는 생성은 trunk 내부 가이드 렌더러 기준이며, Shin VM 추론 런타임 연동은 후속 과제입니다.
4. 외부 SNS 자동 게시 전부를 상용 수준으로 보장하는 단계는 아닙니다.

## 백업 시나리오

### 백업 1: 제품 앱 안에서 범위 축소

- 로그인
- 생성 화면 일부 입력
- 결과/이력/계정 화면 중심 설명

### 백업 2: 독립 프로토타입 사용

- `docs/prototypes/web-flow-prototype.html`
- SC-01 ~ SC-14 전 화면 흐름 설명용
- 실제 API 연동이 아니라 UX 구조 설명용이라고 명확히 말합니다.

## 발표 마무리 문장

- 현재 결과물은 통합이 끝난 발표용 trunk 기준선입니다.
- 핵심 사용자 흐름은 동작하고 검증됐으며, 운영 고도화와 모델 실행 환경 정리는 다음 단계 과제로 남아 있습니다.
