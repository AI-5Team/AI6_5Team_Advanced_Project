# 10. 루트 기준 기능 선별 통합 계획

> **문서 목적**: 현재 루트 저장소를 trunk로 고정한 상태에서 팀원별 산출물을 폴더 단위가 아니라 기능 단위로 선별 통합하기 위한 기준선과 freeze 직전 체크리스트를 정의합니다.

## 1. 고정 원칙

1. 기준선은 현재 루트 저장소입니다.
   - `services/api/app`
   - `apps/web`
   - `packages/contracts`
   - `docs/planning`
2. 통합은 폴더 통째 병합이 아니라 기능 단위 선별 이식으로 진행합니다.
3. `node_modules`, `.env`, 실행 로그, 산출 영상/이미지, 임시 업로드, 로컬 DB/캐시는 통합 대상에서 제외합니다.
4. public contract 변경이 필요하면 `packages/contracts`와 planning 문서를 동시에 갱신합니다.
5. 연구, 비교, 실험, 검증 결과는 모두 문서화해야 완료로 인정합니다.

## 2. 채택 판정 기준

| 축 | 판단 질문 |
|---|---|
| 안정성 | 같은 입력과 같은 실행 조건에서 재현 가능한가 |
| 품질 | 사용자 흐름과 출력 품질이 발표 기준에 도달하는가 |
| 확장성 | trunk 구조 안에서 유지보수와 기능 추가가 쉬운가 |
| 속도 | 구현/운영 복잡도와 실행 시간이 허용 범위인가 |
| 발표 시연 적합성 | 팀 발표에서 설명 가능하고 데모 리스크가 낮은가 |

### 우선순위 하향 조건

- 문서 없는 코드
- 실행 조건이 불명확한 코드
- 재현 불가능한 실험 결과
- trunk의 공용 계약을 깨는 구조

## 3. 통합 대상별 판정

| 대상 | 판정 | trunk 반영 방식 | 현재 상태 |
|---|---|---|---|
| 루트 저장소 | 유지 | source of truth 유지 | 진행 중 |
| 최무영 큐 패턴 | 비교 후 선택 | 루트 background/worker 흐름보다 단순하고 운영 이점이 확인될 때만 흡수 | 보류 |
| 서유종 worker 경계 | 적극 반영 | `services/worker` 내부 모듈 경계 개선에 흡수 | 반영 중 |
| 신유철 모델 실험 저장소 | 독립 유지 | worker adapter 경계로만 연결 | 경계 구현 완료 |
| 이진석 Vite UI | 화면/자산/상태만 포팅 | `apps/web`에 단계형 화면으로 흡수 | 반영 중 |

## 4. 현재 trunk에 반영한 결정

### 4-1. Web

- 프로덕션 웹 앱은 `apps/web` 하나만 유지합니다.
- 로그인은 신규 설계가 아니라 기존 `register/login/me` 계약을 현재 UI 흐름에 연결합니다.
- 인증 세션은 웹 로컬 세션으로 유지하고, 보호 화면은 루트 레이아웃에서 게이트합니다.

### 4-2. API

- `services/api/app` 구조는 유지합니다.
- DB access 기준선은 `app.core.database`의 raw `sqlite3` 경로로 고정합니다.
- SQLAlchemy 기반 초안(`bootstrap`, ORM entities)은 freeze 기준선에서 제외합니다.
- `/api/auth/register`, `/api/auth/login`, `/api/me`를 기준 인증 계약으로 사용합니다.
- `/api/me`는 Bearer token이 없거나 위조된 경우 `401 AUTH_REQUIRED`로 고정합니다.

### 4-3. Worker

- `services/worker/adapters`, `pipelines`, `renderers` 경계는 유지합니다.
- 제품 런타임은 실험 저장소를 직접 import하지 않습니다.
- `adapter_wan2_vace` 경계를 추가해 실험 저장소를 subprocess/환경변수 경계로만 연결합니다.

## 5. 인벤토리 요약

| 출처 | 핵심 강점 | trunk 채택 기준 | 채택 방식 |
|---|---|---|---|
| 사용자 루트 저장소 | API 구조, contracts, planning, Next 앱 기준선 | trunk 유지 | 기준선 |
| 신유철 | Wan2.1-VACE 기반 모델 실험, benchmark 자산 | 실험 저장소 독립 유지 + adapter 경계 | 선택 채택 |
| 이진석 | 단계형 UI 흐름, 화면 구성 완성도 | 현재 Next 앱에 화면 정책과 자산만 포팅 | 선택 채택 |
| 최무영 | 단순한 큐 처리 패턴 | 루트 흐름보다 운영 단순성이 검증될 때만 | 비교 보류 |
| 서유종 | worker 레이어 분리 설계 | 현재 worker 내부 재구성에 우선 반영 | 선택 채택 |

## 6. 실행 순서

1. 인벤토리 작성
2. 계약 정합화 고정
3. 백엔드 안정화
4. 모델 연결
5. UI 포팅
6. 통합 검증
7. Freeze 및 재분배

## 7. Freeze 전 검증 시나리오

### 기능 시나리오

1. 회원가입 → 로그인 → `me` 조회
2. 프로젝트 생성 → 자산 업로드 → 생성 → 결과 조회
3. 결과 확인 → 업로드 보조 또는 게시 fallback
4. 계정 연동 진입 → callback 처리 → 상태 반영
5. 이력 진입 → quick action 재생성
6. 모델 lane 1개를 worker 경유로 smoke run

### 보안 시나리오

1. 잘못된 비밀번호
2. 만료 또는 위조 토큰
3. 미연동 채널 auto publish 시도
4. 지원하지 않는 채널 요청
5. 파일 형식/크기 제한 위반

## 8. Freeze 전 정리 규칙

1. 채택된 구현만 trunk에 남깁니다.
2. 중복 경로는 freeze 전에 정리합니다.
3. 채택/보류/폐기 이유와 남은 리스크를 문서로 남깁니다.

## 9. 업무 재분배 기준선

| 업무 | 주담당 | 범위 |
|---|---|---|
| 모델 개선 | 신유철 | 모델 선택, 실험, 추론 설정, benchmark, worker adapter 개선 |
| 보안 기능 | 이진석 | 업로드 검증, 외부 연동 권한 흐름, 토큰/채널 상태 UX, rate limit, 오류 처리 정책 |
| 로그인 + 관련 보안 | 최무영 | register/login/me 연결, 보호 라우트, 토큰 처리, 인증 예외 정리 |
| UI/UX 개선 | 서유종(구현) + 이진석(디자인 리뷰) | `apps/web` 단계형 화면 구조 정리 |
| 최종 종합 + 발표 | 임창현 | 통합 승인, freeze, 발표 기준선 확정 |

## 10. 현재 남은 리스크

1. 인증은 UI와 `me` 검증까지 연결됐지만, 프로젝트/채널 데이터는 아직 demo-state 중심입니다.
2. Wan2.1-VACE adapter는 경계만 고정했고, 실제 benchmark lane 활성화는 환경 준비가 필요합니다.
3. 큐 구조는 비교 전이므로 Redis/RQ 채택 여부는 아직 확정되지 않았습니다.
