| Must | 템플릿 기반 숏폼 생성 | 발표 핵심 기능 |

| Must | `instagram` 1채널 업로드 흐름 | “운영형 서비스” 증명 |
| Must | 결과 상태 조회와 실패 분리 | 비동기 파이프라인 필수 |
| Must | 공동 숏폼 프로토타입 비교 | 팀 운영 방식상 필수 |

| Should | 계정 연동 UX 고도화 | 중장년층 이탈 방지 |

## 0. 프로젝트 한 줄 정의

> **“사진 몇 장과 몇 번의 선택만으로, 소상공인이 SNS 광고용 숏폼과 게시글을 만들고 실제로 업로드 또는 업로드 보조까지 이어갈 수 있게 하는 운영형 서비스”**

핵심 문제는 “광고를 예쁘게 만드는 것”이 아니라, **소상공인이 광고를 꾸준히 올릴 수 없는 구조**에 있습니다.  
따라서 본 서비스는 편집기 경쟁이 아니라 `입력 간소화 → 템플릿 생성 → 채널 변환 → 게시`의 연결성을 제품 가치로 둡니다.
Phase 1의 핵심 타깃은 카페/음식점 중심의 지역 소상공인, 특히 디지털 도구에 익숙하지 않은 중장년층 운영자입니다.

## 1. 시스템 아키텍처

```text
+------------------------------------------------------+
| Layer 0: Product Policy Router                       |
| 업종/목적/스타일/채널에 따라 허용 템플릿과 fallback 결정 |
+------------------------------------------------------+
| Intake Layer                                         |
| 사용자 입력 정규화, 업종/지역/사진/채널/옵션 검증      |
+------------------------------------------------------+
| Planning Layer                                       |
| 템플릿 선택, 카피 규칙 선택, 렌더링 파라미터 생성      |
+------------------------------------------------------+
| Generation Layer                                     |
| 이미지 전처리, 캡션 생성, 숏폼 렌더링, 게시글 생성      |
+------------------------------------------------------+
| Distribution Layer                                   |
| 채널별 포맷 변환, 업로드, 업로드 보조, 예약 발행(Phase 2), 재시도 처리 |
+------------------------------------------------------+
| Evidence Layer                                       |
| 상태 로그, 실험 로그, 생성 결과 메타데이터, 실패 기록  |
+------------------------------------------------------+
```

### Layer 0. Product Policy Router

- 업종, 목적, 스타일, 채널 조합으로 허용 템플릿을 제한합니다.
- 지원하지 않는 채널/옵션은 early fallback 합니다.
- “같은 입력인데 팀원마다 다른 제품을 만든다”는 상황을 방지하는 기준층입니다.

### Intake Layer

- 사용자 입력 검증
- 파일 형식/크기 검증
- 지역 문자열 정규화
- 프로젝트 생성과 상태 초기화

### Planning Layer

- 템플릿 후보 선정
- 카피 규칙 선택
- 영상 컷 구성 결정
- quick action change set 반영

### Generation Layer

- 이미지 자동 보정
- 배경 제거
- 카피 생성
- 숏폼 렌더링
- 게시글용 이미지 생성

### Distribution Layer

- 채널별 포맷 후처리
- 즉시 업로드
- 예약 발행
- 실패 재시도

### Evidence Layer

- 실험 결과 저장
- 생성 상태/실패 원인 로그
- 사용자 테스트 기록
- ADR/실험 문서 링크 관리

## 2. 핵심 서비스 루프

```text
사용자 입력
  -> 프로젝트 생성
  -> 원본 이미지 업로드
  -> 전처리 작업 큐 등록
  -> 템플릿/카피 계획 수립
  -> 숏폼/게시글 생성
  -> 결과 검수 및 빠른 수정
  -> 업로드 또는 업로드 보조
  -> 성공/실패 기록
  -> 재사용 또는 재생성
```

### 루프에서 중요한 점

1. 생성과 게시는 같은 서비스 루프 안에 있어야 합니다.
2. 생성 실패와 업로드 실패는 서로 다른 실패로 취급합니다.
3. 사용자는 정교한 편집보다 빠른 수정과 재생성을 사용합니다.

## 3. 제품 불변 원칙

### 3-A. 프롬프트를 사용자에게 요구하지 않습니다

사용자는 자연어 명령을 길게 쓰지 않습니다.  
서비스가 선택형 입력을 구조화된 생성 파라미터로 바꿉니다.

### 3-B. 템플릿 기반 생산성을 우선합니다

영상을 완전히 새로 생성하는 방식보다, **예측 가능한 템플릿 조합**을 우선합니다.

### 3-C. 생성 결과는 바로 게시 가능한 수준이어야 합니다

숏폼, 게시글, 캡션은 저장용 샘플이 아니라 실제 업로드 가능한 결과물을 목표로 합니다.

### 3-D. 지역성은 옵션이 아니라 기본값입니다

지역 정보가 들어오면 결과물에 반영되어야 합니다.

### 3-E. 실패를 숨기지 않습니다

전처리 실패, 렌더 실패, 업로드 실패는 모두 다른 상태값과 로그로 남겨야 합니다.

### 3-F. 실험은 반드시 문서화합니다

팀 운영 규칙상 과정과 결과 기록은 선택이 아니라 필수입니다.

## 4. Phase 1 제품 범위와 fallback

### Phase 1 핵심 범위

- 업종 2개: 카페, 음식점
- 스타일 3개: 기본, 친근함, 하찮고 웃김
- 템플릿 4개
- 사진 촬영 가이드와 품질 경고
- 숏폼 5~7초
- 게시글 이미지 1종 이상
- 캡션 후보 3개
- `instagram` 1채널 업로드 또는 업로드 보조
- 결과물은 생성 즉시 이력에 자동 저장

Phase 1 기준선에서는 `예약 발행`과 `추천 템플릿 갱신`을 데모 필수선에 포함하지 않습니다.

### fallback 원칙

| 실패 지점 | fallback |
|---|---|
| 배경 제거 실패 | 보정된 원본 사용 |
| 렌더링 실패 | 다른 템플릿으로 재시도 |
| 자동 업로드 실패 | 업로드 보조 모드 전환 |
| 특정 채널 미지원 | 지원 채널만 노출 |
| 스타일 결과 품질 저하 | 기본 스타일로 자동 fallback |

## 5. 기술 스택과 선택 기준

| 역할 | 권장 스택 | 선택 이유 |
|---|---|---|
| Web | Next.js + TypeScript | 폼/상태/UI 구성에 적합 |
| API | FastAPI | Python AI 파이프라인과 연동 용이 |
| Worker | Python + Redis queue | 무거운 작업 분리 |
| DB | PostgreSQL | 상태, 이력, 메타데이터 저장 |
| Cache/Queue | Redis | 비동기 처리 |
| Storage | GCS/S3 호환 | 이미지/영상 저장 |
| Template Spec | JSON/YAML + versioning | 코드와 규칙 분리 |

### 5-A. 생성 모델 후보군과 선택 원칙

Phase 1에서는 생성 모델을 단일 벤더/단일 모델로 미리 고정하지 않습니다.  
카피 생성, 구조화 JSON 출력, 이미지 이해/OCR, 로컬 실행 가능성을 기준으로 후보군을 유지하고 비교 후 기본값을 확정합니다.

| 후보군 | 분류 | 우선 검증 용도 | 채택 이유 | 유의점 |
|---|---|---|---|---|
| `Gemma 4` | 오픈 멀티모달 모델 | 카피 생성, 구조화 출력, 이미지 이해, 로컬/온프렘 실험 | Google이 `2026-04-02` 공개한 오픈 모델 계열로, Apache 2.0 라이선스, 긴 컨텍스트, 구조화 출력/에이전트 워크플로우 적합성이 있어 연구 가치가 큼 | 한국어 광고 카피 톤, 실제 추론 지연, 서빙 난이도는 별도 벤치마크 필요 |
| 관리형 상용 멀티모달 모델 | API 후보군 | 시연용 기본 품질선, 안정적인 실패 처리 | 초기 MVP 데모 안정성과 운영 편의성이 높음 | 비용, 벤더 종속성, rate limit 관리 필요 |
| 경량 관리형 모델 | API 후보군 | 캡션 후보 다량 생성, fallback 생성 | 빠른 응답과 비용 절감에 유리 | 품질 하한선 미달 시 메인 기본값으로 사용 불가 |

### 5-B. 모델 선정 원칙

1. 발표 안정성이 최우선인 기본 경로는 관리형 모델을 우선 검토합니다.
2. `Gemma 4`는 로컬 실행, 오픈 라이선스, 구조화 출력 검증을 위한 핵심 오픈 후보로 유지합니다.
3. 기본 모델 교체 전에는 반드시 `docs/experiments/`에 비교 기록을 남기고, 기준선 변경 시 `docs/adr/`에 결정 사유를 남깁니다.
4. 모델별 차이는 최소한 `한국어 카피 품질`, `JSON schema 준수율`, `OCR/이미지 이해`, `지연 시간`, `운영 복잡도`로 비교합니다.

### 선택 기준

1. 프론트와 AI 모듈의 구현 언어 경계를 분명히 합니다.
2. 비동기 작업이 분리되어야 합니다.
3. 템플릿 규칙은 코드보다 데이터 파일 중심으로 관리합니다.

### 경로별 설명

| 경로 | 무엇을 두는 폴더인가 |
|---|---|
| `apps/web` | 사용자가 직접 보는 화면, 입력 폼, 결과 확인 화면, polling UI, 계정 연동 UX를 두는 프론트엔드 폴더입니다. |
| `services/api` | 프로젝트 생성, 상태 조회, 결과 조회, 게시 요청 검증, 큐 등록 같은 REST API 서버 코드를 두는 폴더입니다. |
| `services/worker` | 전처리, 카피 생성, 렌더링, 자동 업로드, 예약 실행처럼 오래 걸리는 비동기 작업을 처리하는 워커 폴더입니다. |
| `packages/contracts` | 프론트/API/워커가 같이 써야 하는 enum, 상태값, payload schema, 공통 타입을 두는 공유 계약 폴더입니다. |
| `packages/template-spec` | 템플릿 JSON, 카피 규칙, 스타일 규칙처럼 생성 파이프라인이 읽는 데이터 중심 규칙을 두는 폴더입니다. |
| `packages/shared-assets` | 공용 폰트, 아이콘, 오버레이, 템플릿 공통 시각 자산을 두는 폴더입니다. |
| `docs/planning` | 현재 서비스 기획, 범위, 정책, 계약 문서를 두는 기준 문서 폴더입니다. |
| `docs/experiments` | 실험/리서치/스파이크의 과정과 결과를 기록하는 폴더입니다. 실험은 반드시 문서화합니다. |
| `docs/adr` | Architecture Decision Record를 두는 폴더입니다. 주요 기술 선택과 변경 이유를 남깁니다. |
| `docs/daily` | 일일 진행 상황, 결정 사항, 막힌 이슈를 짧게 기록하는 운영 로그 폴더입니다. |
| `docs/testing` | 테스트 시나리오, 테스트 결과, QA 체크리스트를 두는 검증 기록 폴더입니다. |
| `samples` | 데모용 입력 이미지, 예시 결과물, QA 재현용 샘플 데이터를 두는 폴더입니다. |
| `infra` | Docker, 환경 변수 템플릿, 배포 설정, 인프라 실행 파일을 두는 폴더입니다. |
| `scripts` | 로컬 개발, 검증, 데이터 준비, 운영 보조 자동화 스크립트를 두는 폴더입니다. |

### 모듈 경계

| 모듈 | 하는 일 | 하지 않는 일 |
|---|---|---|
| web | 입력/결과/상태 표시 | 렌더링/업로드 처리 |
| api | 검증, 프로젝트 상태, 큐 등록 | 무거운 생성 작업 |
| worker | 전처리, 생성, 업로드 | 사용자 세션 처리 |
| contracts | 상태값, enum, payload 정의 | 구현 로직 |
| template-spec | 템플릿 규칙 저장 | 렌더링 실행 |
| shared-assets | 공용 시각 자산 제공 | 상태값/비즈니스 규칙 정의 |

## 7. 운영 가정과 제약

### 초기 도메인 가정

- 업종: 카페/음식점
- MVP 게시 채널: 인스타그램
- 확장 후보 채널: 유튜브 쇼츠, 틱톡
- 지역 중심 매장 홍보
- 한국어 광고 카피 기준

### 제약

- Phase 1에서 자동 게시를 보장하는 채널은 `instagram` 1개로 제한합니다.
- `youtube_shorts`, `tiktok`은 계약 enum에는 존재하지만, Phase 1에서는 `experimental` 또는 `업로드 보조 전용`으로 취급합니다.
- `예약 발행`은 Phase 2 확장 기능으로 분리합니다.
- `추천 템플릿 갱신`은 Phase 3 연구 과제로 분리합니다.
- 5~7초 숏폼으로 길이를 제한합니다.
- 생성형 비디오는 선택 실험이며 MVP 필수 의존성이 아닙니다.

## 8. 평가 기준 및 Go/No-Go

| 지표 | 목표치 | 우선순위 |
|---|---|---|
| 문제 적합성 인터뷰 | 타깃 사용자 3명 이상에서 핵심 pain point 확인 | 최우선 |
| 첫 결과물 생성 시간 | 3분 이내 | 최우선 |
| 생성 성공률 | 90% 이상 목표 | 높음 |
| 업로드/업로드 보조 성공률 | `instagram` 1채널 기준 시연 가능 | 최우선 |
| 스타일 식별성 | 3스타일 구분 가능 | 높음 |
| 지역 반영 적합도 | 사용자 주관 평가 4.0/5 이상 목표 | 높음 |
| 문서화 누락 | 0건 | 필수 |

### Go/No-Go 기준

- 숏폼 생성이 안정적으로 재현됩니다.
- `instagram` 게시 또는 게시 보조 흐름이 시연 가능합니다.
- 공동 개발 숏폼 결과를 하나의 통합안으로 묶을 수 있습니다.
- 최소 3명의 타깃 사용자 인터뷰에서 문제 정의가 완전히 빗나가지 않았음이 확인됩니다.
- 실험 로그와 실패 문서가 누락되지 않습니다.
