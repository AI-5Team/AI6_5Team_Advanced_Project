# 06. 평가, 테스트, 운영 기준

> **문서 목적**: “만들었다”와 “쓸 수 있다”를 구분하기 위한 KPI, 테스트 시나리오, 운영 로그 기준을 정의합니다.

## 1. 핵심 KPI

| 지표 | 목표 |
|---|---|
| 문제 적합성 인터뷰 | 타깃 사용자 3명 이상 공통 pain point 확인 |
| 첫 결과물 생성 시간 | 3분 이내 |
| 생성 성공률 | 90% 이상 목표 |
| 게시 흐름 성공 | 1개 채널 기준 `자동 업로드` 또는 `업로드 보조` 완료 가능 |
| 계정 연동 완료율 | 타깃 사용자 테스트에서 80% 이상 완료 목표 |
| 업로드 보조 완료율 | 업로드 보조 진입 케이스의 80% 이상이 완료 확인까지 도달 |
| 스타일 식별성 | 3스타일 구분 가능 |
| 지역 적합도 | 4.0/5 이상 목표 |
| 문서화 누락 | 0건 |

### 모델 후보 비교 KPI

모델 후보 비교는 품질 체감만으로 결정하지 않고 아래 항목을 함께 기록합니다.

| 항목 | 기준 |
|---|---|
| 한국어 카피 적합도 | 업종/목적/스타일에 맞는 표현을 사람 검수에서 통과 |
| JSON schema 준수율 | 계약된 필드를 누락 없이 반환 |
| 이미지 이해/OCR 정확도 | 메뉴명, 가격, 지역명 추출 오류가 허용 범위 이내 |
| 평균 지연 시간 | 데모 허용 시간 안에 응답 완료 |
| 재현성 | 동일 입력에서 과도한 품질 편차가 없음 |
| 운영 복잡도 | 인증, 비용, 배포, 로컬 실행 난이도가 팀 수용 범위 안 |

## 2. 문제 검증 기준

구현이 맞는 방향인지 판단하려면 “만들기 전에” 아래 질문이 최소 확인되어야 합니다.

1. 직접 릴스/쇼츠를 만드는 것이 실제로 어렵습니까
2. 편집보다 업로드/예약까지 연결되는 흐름이 더 필요합니까
3. 자동 생성된 지역 문구가 실제 홍보에 도움이 된다고 느낍니까

### 최소 기준

- 타깃 사용자 3명 이상 인터뷰
- 2명 이상이 “직접 만들기 어렵다”를 명시적으로 말함
- 2명 이상이 “게시까지 연결되면 좋겠다”를 말함
- 2명 이상이 “지역성 반영이 의미 있다” 또는 “있으면 좋다”고 응답
- 계정 연동 단계에서 과반이 별도 도움 없이 완료 가능

### 모델 후보 비교 실험 문서화 규칙

- `Gemma 4`를 포함한 모델 비교는 반드시 `docs/experiments/EXP-번호-주제.md` 형식으로 남깁니다.
- 문서에는 후보 모델명, 입력 샘플, 출력 예시, 점수표, 실패 사례, 최종 판단을 함께 적습니다.
- 기본 모델을 바꾸는 결정은 `docs/adr/ADR-번호-결정사항.md`로 승격합니다.

## 3. 사용자 테스트 시나리오

### TS-01 카페 신메뉴

- 업종: 카페
- 목적: 신메뉴
- 스타일: 친근함
- 기대: 숏폼 + 게시글 + 캡션 생성 후 게시 가능

### TS-02 음식점 할인

- 업종: 음식점
- 목적: 할인/행사
- 스타일: 기본
- 기대: 할인형 메시지와 CTA가 드러나는 결과물 생성

### TS-03 빠른 수정

- 기존 프로젝트 열기
- “문구 더 짧게” 적용
- 재생성
- 차이가 눈에 보여야 함

### TS-04 계정 연동 및 업로드 보조

- `instagram` 계정 연동 진입
- 자동 업로드 실패 또는 미지원 채널 선택 시 업로드 보조 전환
- 사용자가 수동 업로드 완료 확인까지 수행

## 4. 결과물 품질 체크리스트

아래 항목은 디자이너/개발자 공동 체크 기준으로 사용합니다.

1. 자막이 안전 영역을 벗어나지 않는가
2. 핵심 문구가 최소 1.2초 이상 노출되는가
3. 상품/메뉴 피사체가 명확히 보이는가
4. CTA가 마지막 씬에서 식별 가능한가
5. 스타일 차이는 나지만 제품 정보 전달은 유지되는가
6. 지역명/혜택/가격 중 최소 1개가 즉시 인지되는가

## 5. 운영 로그 최소 항목

- user_id
- generation_run_id

- preprocess_status

- assist_status

## 6. 실패 사례 분류

| 구분 | 예시 |
|---|---|
| 입력 실패 | 파일 형식/크기 오류 |
| 전처리 실패 | 배경 제거 실패 |
| 생성 실패 | 카피/렌더 오류 |
| 게시 실패 | 권한/포맷/API 오류 |

## 7. Go/No-Go

### Go

- 최소 1채널 흐름이 발표 환경에서 반복 재현됨
- 계정 연동 또는 업로드 보조 중 하나로 사용자가 게시 완료까지 도달함
- 숏폼 스타일 차이가 식별됨
- 문제 적합성 인터뷰 기준을 통과함
- 실험 로그와 결과 스크린샷이 남아 있음

### No-Go

- 숏폼 생성이 불안정해 발표 직전까지 재현 보장이 없음
- 게시 흐름이 전혀 이어지지 않음
- 팀원별 구현물이 통합 기준 없이 분산됨
