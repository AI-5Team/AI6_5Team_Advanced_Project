- 대신 `template/hybrid 광고 패키징 + upload assist` 본선은 실제로 웹에서 끝까지 동작하는 상태까지 왔습니다.
- 따라서 이번 회의의 핵심은 `무엇을 더 연구할지`보다 **지금 이 제품을 어떤 이름과 약속으로 팀이 밀 것인지**를 정하는 것입니다.

## 2. 지난 회의 이후 달라진 점

### 2.1 방향 재판정이 공식화됐습니다

- `EXP-259`, `EXP-260`, `EXP-261`, `EXP-262`를 거치며 아래 판단이 거의 일치했습니다.
  1. 최근 실험선은 일부 연구 드리프트가 맞습니다.
  2. 특히 `LTX/Sora 미세 OVAT 반복`은 지금 당장 본선 우선순위가 아닙니다.
  3. 현재 자원 기준으로 현실적인 본선은 `template/hybrid + upload assist`입니다.

### 2.2 로컬 영상 후보는 다시 좁혀졌습니다

- `docs/planning/09_EXECUTION_PRIORITY_AND_MODEL_CANDIDATES.md` 기준으로 현재 로컬 후보는 `Wan 2.1` 일부 경로만 남겼습니다.
- `Wan 2.2`, `LTX 추가 OVAT`, `Sora single-photo 반복`은 현재 우선순위에서 내렸습니다.
- frontier 후보는 `Seedance 2.0`, `Veo 3.1`을 future track으로만 올려둔 상태입니다.

### 2.3 본선 웹은 사용자 테스트용 표면까지 다시 정리됐습니다

- 메인 웹은 복잡한 내부용 UI가 아니라, `사진 -> 목적 -> 결과 확인 -> 업로드 준비` 흐름 중심으로 다시 정리됐습니다.
- 라이트/다크 모드, 대비 문제, 결과 placeholder 문제도 최근 세션에서 정리됐습니다.

### 2.4 가장 중요한 변화: 본선 E2E가 실제로 검증됐습니다

- 이번 세션에서 `web -> api -> worker -> media -> web preview`를 실제로 닫았습니다.
- 즉 지금은 “이론상 된다”가 아니라, **실제 버튼을 누르면 결과 video/post가 다시 웹으로 돌아오는 상태**입니다.

## 3. 지금 시점에서 확인된 사실

### 3.1 사실

1. 현재 저장소가 가장 안정적으로 만드는 것은 `template/hybrid` 기반 결과물입니다.
2. 현재 웹은 `NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:8000` 기준으로 FastAPI를 직접 바라봅니다.
3. 실제 샘플 이미지 기준으로
   - `POST /api/projects`
   - `POST /assets`
   - `POST /generate`
   - `GET /status`
   - `GET /result`
   흐름이 성공했고, `video.mp4`, `post.png` 응답도 정상 확인됐습니다.
4. 브라우저 자동화 기준으로도 `콘텐츠 만들기` 클릭 후 결과 화면의 원본 링크가 실제 FastAPI media 경로를 가리키는 것까지 확인됐습니다.

### 3.2 아직 사실로 말할 수 없는 것

1. `완전 자동 광고형 동적 숏폼`이 현재 자원으로 출시 가능하다고는 말할 수 없습니다.
2. `Seedance`, `Veo`, `Kling`, `Runway` 등이 당장 우리 본선을 해결한다고 단정할 수 없습니다.
3. 비용/쿼터/API 접근성 같은 외부 변수는 이번 회의자료에서 확정 사실처럼 말하지 않는 것이 맞습니다.

문제는 기술력이 부족해서가 아니라, **현재 자원/모델/제어성 기준으로 팀이 약속할 수 있는 수준을 넘기 때문**입니다.

## 6. 이번 회의에서 반드시 결정해야 할 것

### 결정 1. 제품 정의를 공식적으로 줄일 것인가

선택지:

1. 여전히 `완전 자동 광고 숏폼 생성`을 전면 약속으로 유지한다
2. 당분간은 `하이브리드 광고 패키징 + 업로드 보조`를 공식 본선으로 둔다

제안:

- **2번을 공식 본선으로 두는 것이 맞습니다.**

### 결정 2. 영상 연구선을 어떻게 둘 것인가

선택지:

1. 계속 본선과 같은 레벨로 밀어붙인다
2. future candidate track으로 낮추고, 본선과 분리한다

제안:

- **2번이 맞습니다.**
- 즉 영상 연구는 아예 버리는 것이 아니라, `본선과 섞지 않고 별도 후보선`으로 관리하는 방식이 맞습니다.

### 결정 3. 다음 1주 우선순위를 어디에 둘 것인가

선택지:

1. 모델 실험 확대
2. 본선 사용자 테스트 준비
3. 실제 게시 연동

제안:

- **2번이 먼저입니다.**
- 게시 연동은 그 다음이고, 모델 확대는 그보다 뒤입니다.

## 7. 제가 팀에 권하는 결론

### 결론 A. 지금부터 본선은 이렇게 부릅니다

- `template/hybrid 광고 제작 + upload assist`

### 결론 B. 영상 연구는 이렇게 다룹니다

- `로컬/상용 영상 생성 후보선은 future track`
- 본선 merge 기준은 오직 아래 4개입니다.
  1. 원본 보존
  2. 광고형 motion
  3. repeatability
  4. 비용/시간 현실성

### 결론 C. 팀 리소스는 이렇게 씁니다

- 제품 구현/사용자 테스트: 앞으로 당분간 메인
- 영상 모델 실험: 질문이 새로 생길 때만 제한적으로

## 8. 회의 중 예상되는 질문과 답

### Q1. 그러면 영상 생성은 포기한 것인가

- 아닙니다.
- 다만 **본선 약속에서 잠시 내린 것**입니다.
- 다시 말해, 영상 생성은 `버림`이 아니라 `future candidate track`입니다.

### Q2. 왜 지금 연구를 더 안 늘리나

- 지금 병목은 “더 많은 후보를 모르는 것”이 아니라, **팀이 무엇을 만들고 있다고 말할지 아직 고정되지 않은 것**이기 때문입니다.
- 제품 정의가 먼저 고정돼야 연구도 덜 샙니다.

### Q3. 지금 사용자 테스트를 해도 되나

- 네, 됩니다.
- 다만 테스트 질문은 “AI 영상 품질이 감탄스러운가”가 아니라,
  - 이 흐름을 이해하는가
  - 결과를 유용하다고 느끼는가
  - 업로드 준비까지 이어갈 수 있는가
  를 봐야 합니다.
