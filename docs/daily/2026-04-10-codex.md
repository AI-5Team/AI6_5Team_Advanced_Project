## 결과

1. `image-to-video -> edit` 2단계는 완전히 무의미하진 않았습니다.
2. 적어도 `same_shot_push_in_motion`은 source보다 motion을 실제로 더 만들었습니다.
3. 하지만 motion이 증가한 만큼 preserve loss도 같이 크게 증가했습니다.
4. 즉 `edit`는 trade-off를 없애는 해법이 아니라, `보존성을 일부 희생해서 motion을 조금 회복하는 수단`에 더 가깝습니다.
5. 외부 레퍼런스 정리까지 합치면, 지금 필요한 건 `raw prompt digging` 확대보다 `reference teardown + objective split`입니다.
6. 그리고 그 정리를 backlog 관점으로 바꾸면, 다음 액션은 `hook/copy`, `평가 기준 분리`, `입력 구조 변경` 쪽으로 모입니다.
7. 이제 레퍼런스 수집 자체도 `템플릿 적합도 판정` 체계 안에서 누적할 수 있게 됐습니다.
8. 그리고 외부 AI ad platform 분석까지 합치면, 본선 제품 판단 기준도 `영상 품질 1편`보다 `self-serve 템플릿 시스템` 쪽으로 더 선명해졌습니다.
9. `EXP-98`까지 합치면, 이제 그 self-serve 템플릿 시스템을 현재 template-spec 위에 어떻게 얹을지에 대한 최소 구조도 잡혔습니다.
10. `EXP-99`까지 합치면, 이제 그 구조 위에 실제 hook 후보까지 데이터로 누적하기 시작했습니다.
11. `EXP-100`, `EXP-101`까지로 그 hook 후보가 실제 모델 제어 레버로도 기능한다는 신호를 확인했습니다.
12. `EXP-102`까지로는 모델별 흔들림 패턴도 드러났습니다.
13. `EXP-103`~`EXP-105`까지로는 각 모델에 어떤 constraint가 먹히고 어떤 부분이 계속 새는지도 더 구체적으로 드러났습니다.

## 실패/제약

1. 스킬 번들 CLI의 `edit` 경로는 현재 SDK parsing bug 때문에 그대로는 쓰기 어려웠습니다.
2. 이번 결과도 baseline Sora나 manual Veo 수준의 motion까지는 올라가지 못했습니다.
3. `same-shot` edit는 가능하지만, 아직 production 후보라고 보기엔 preserve loss가 큽니다.
4. 유튜브 레퍼런스는 제목/공개 페이지 수준의 1차 확인이라, shot-by-shot 분해는 별도 수동 시청이 더 필요합니다.
5. 따라서 `EXP-95`는 레퍼런스 전체 복제가 아니라, 현재 시점에서 신뢰할 수 있는 수준의 패턴 추출 문서로 봐야 합니다.
6. `EXP-96`의 예시 점수는 초기 버전이므로, 실제 레퍼런스를 더 쌓으면서 보정이 필요합니다.
7. `EXP-97`도 공개 마케팅 페이지 기준 분석이라, 실제 내부 editor/UX 전체를 확인한 것은 아닙니다.
8. `EXP-98`의 slot-layer-map은 draft이므로, 아직 worker/web가 읽지 않습니다.
9. `EXP-99`의 hook pack도 아직 runtime 미사용 초안입니다.
10. `EXP-100`, `EXP-101`은 모두 single-run 기준이라 repeatability 확인이 아직 필요합니다.
11. repeatability spot check는 했지만, 아직 샘플 수와 반복 수는 작습니다.
12. `EXP-103`~`EXP-105`도 모두 단일 샘플/단일 run 위주라서, 절대 결론이라기보다 prompt tuning 방향 확인에 가깝습니다.
