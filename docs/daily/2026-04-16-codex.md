## 실제 변경

### 1. 상단 네비 단순화

- 브랜드명과 3개 진입점만 남겼습니다.
- `사용자 테스트`, 부가 설명 문구, 메뉴별 캡션은 제거했습니다.

### 2. 메인 첫 화면 구조 재설계

- 처음 시도했던 `좌측 비주얼 / 우측 입력 레일` 구조는 레퍼런스 이해가 부족했다고 판단해 버렸습니다.
- 메인 화면을 `전체 폭 히어로 + 하단 실행 블록` 구조로 다시 짰습니다.
- 히어로는 큰 문장 1개와 대표 이미지에 집중하고, 실행 블록은 `사진 / 목적 / 가게 / 시작` 4개 그룹만 남겼습니다.
- 목적/업종 선택은 라벨만 보이게 줄였고, 첫 화면에서 다음 액션이 살짝 보이도록 히어로 높이를 조정했습니다.

### 3. 전역 팔레트 흑백화

- 루트 색상 토큰을 모두 흑백 계열로 교체했습니다.
- `button`, `badge`, `workspace`, `history`, `accounts` 화면의 상태 강조색도 모두 회색/검정 계열로 맞췄습니다.

### 4. 샘플 이미지 흑백화

- 샘플 SVG 생성 마크업에서 색 면, 원형 장식 요소를 제거했습니다.
- 샘플 이미지는 검정/회색/흰색 기준 시안만 남겼습니다.

### 5. second pass: 같은 날 한 번 더 정리

- 첫 패스 이후에도 메인 화면이 여전히 말이 많고, `history / accounts`는 메인과 다른 제품처럼 보인다고 판단했습니다.
- 그래서 두 번째 패스에서는 색상만 바꾸는 수준이 아니라, 보조 화면까지 포함한 표면 처리와 문장량을 다시 줄였습니다.
- 구체적으로는 아래를 적용했습니다.
  - 전역 토큰에서 남아 있던 warm tint와 녹색/주황 상태색을 전부 제거하고, 검정/회색만 남겼습니다.
  - `workspace-overview`, `workspace-rail`, `workspace-main`, `project-row`, `account-row`, `workspace-kpi`의 그림자와 카드 느낌을 더 걷어냈습니다.
  - 통계 영역은 3개 카드가 아니라 얇은 stat strip처럼 보이도록 바꿨습니다.
  - 메인 히어로 문구와 실행 블록 제목을 한 단계 더 짧게 줄였습니다.
  - 히스토리/채널 화면 copy도 더 운영 화면답게 짧게 정리했습니다.

### 6. third pass: 컬러 팔레트와 라이트/다크 모드 복구

- 흑백은 중간 점검용이었고, 최종 방향은 아니라고 판단했습니다.
- 그래서 다시 레퍼런스 3개를 화면 기준으로 읽어, 아래 공통 구조만 가져왔습니다.
  - 어두운 히어로 영역
  - 청록 계열 발광 포인트
  - 코럴/앰버 계열 보조 강조
  - 밝은 본문 표면 또는 진한 다크 패널
- 이 관찰을 바탕으로 palette를 다시 잡았습니다.
  - light: `icy blue background + cyan accent + warm coral secondary`
  - dark: `ink navy surface + cyan accent + warm coral secondary`
- 구현은 전역 토큰을 다시 theme variable로 묶고, `html[data-theme]` 기반으로 light/dark를 분기하는 방식으로 넣었습니다.
- 상단 네비에는 `ThemeToggle`을 추가했습니다.
- 검증을 위해 `?theme=light`, `?theme=dark` query도 읽도록 해 deterministic screenshot 경로를 확보했습니다.
- 샘플 SVG도 다시 컬러 시안으로 바꿨고, 결과 미리보기/업로드 준비 패널도 theme variable을 따라가도록 수정했습니다.

### 7. fourth pass: 결과 확인 placeholder 흑백 잔재 제거

- `결과 확인` 섹션의 생성 전 placeholder 이미지만 여전히 흑백으로 보이는 문제를 확인했습니다.
- 원인은 기존 reference pass CSS의 `.studio-result-placeholder__visual img { filter: grayscale(1) ... }` 규칙이 남아 있었는데, 히어로와 달리 theme override가 없어 그대로 적용되던 것이었습니다.
- theme pass 끝단에 `--theme-result-image-filter`를 추가하고, `studio-result-placeholder__visual img`를 그 변수로 다시 덮어 컬러 기준으로 복구했습니다.
- 실제 결과 컴포넌트(`ResultMediaPreview`)의 게시 이미지에는 별도 grayscale filter가 없었고, 이번 수정은 placeholder 전용 잔재 정리에 해당합니다.

### 8. local preview recovery: stale Next static manifest

- 컬러 복구 직후 local preview가 거의 무스타일 상태로 보이는 문제를 확인했습니다.
- 조사 결과 CSS 문법 문제가 아니라, 기존 `next start` 프로세스가 이전 빌드의 HTML manifest를 계속 내보내고 있었습니다.
- 당시 HTML은 `/_next/static/css/37f2...css`를 참조했지만, 실제 최신 빌드 산출물은 `4895998...css`였고, 그 결과 CSS 요청이 `400 Bad Request`로 떨어졌습니다.
- 로컬 `3010` 포트의 기존 프로세스를 종료한 뒤 `npm run start`를 재기동해 최신 build manifest와 static asset hash를 다시 맞췄습니다.
- 재기동 이후에는 HTML이 실제 파일과 같은 `4895998ad08f3f14.css`를 참조하고, 해당 CSS 응답도 `200`으로 복구된 것을 확인했습니다.

### 9. light/dark contrast fix for helper copy and field labels

- 사용자 피드백 기준으로 light mode에서는 히어로 보조 문구와 `결과 확인` footer 문구가 잘 보이지 않았고, dark mode에서는 `지역 이름 / 위치 설명 / 게시글 문구 / 해시태그 / 썸네일 문구 / 최근 작업` 계열 라벨과 설명이 거의 읽히지 않는 문제를 확인했습니다.
- 원인은 theme pass 이후에도 일부 selector가 여전히 기존 `var(--muted)`를 사용하고 있었고, 반대로 어두운 배경 위 문구까지 공통 `var(--theme-muted)`로 덮여 대비가 무너진 데 있었습니다.
- 수정은 두 갈래로 정리했습니다.
  - light/dark 공통 입력 라벨, 섹션 설명, 최근 작업 보조 텍스트는 `var(--theme-muted)`로 통일
  - 어두운 히어로/placeholder footer 위 문구는 `var(--theme-hero-meta)`로 분리
- 이 변경으로 light/dark 모두 문구 대비가 다시 살아났고, 보조 문구와 입력 라벨이 읽히는 상태로 복구됐습니다.

## 10. 본선 end-to-end 검증

### 목적

- 현재 본선으로 정리한 `template/hybrid + upload assist` 경로가 실제 웹에서 끝까지 이어지는지 확인합니다.
- 검증 범위는 `apps/web -> services/api(8000) -> worker generation -> media 파일 -> web 결과 미리보기`입니다.

### 확인한 전제

- `apps/web/.env.local` 기준 웹은 `NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:8000`을 사용합니다.
- 따라서 local preview `3010`에서 버튼을 눌렀을 때, 실제 생성 호출은 Next mock 경로가 아니라 FastAPI `8000`으로 나가야 합니다.

### 수행 내용

1. FastAPI `8000` 상태 확인
   - 처음 확인 시 `http://127.0.0.1:8000/api/projects`는 연결 거부 상태였습니다.
   - 첫 기동 시도는 `services/api` 기준 실행이라 `ModuleNotFoundError: No module named 'services'`로 실패했습니다.
   - 이후 `PYTHONPATH=E:\\Codeit\\AI6_5Team_Advanced_Project`를 명시해 다시 띄웠고, `GET /api/projects`가 `200`으로 복구됐습니다.

2. API 레벨 실제 생성 검증
   - 샘플 이미지 `docs/sample/음식사진샘플(맥주).jpg`로
     - `POST /api/projects`
     - `POST /api/projects/{id}/assets`
     - `POST /api/projects/{id}/generate`
     - `GET /status`
     - `GET /result`
     를 순서대로 호출했습니다.
   - 실제 결과:
     - `projectStatus=generated`
     - step 상태 `preprocessing/copy_generation/video_rendering/post_rendering/packaging=completed`
     - `video.mp4` 응답 `200`
     - `post.png` 응답 `200`
     - `rendererMotionMode=static_concat`
   - 즉 API와 worker 기준으로는 현재 본선 산출물이 실제 파일까지 닫힙니다.

3. 브라우저 레벨 실제 생성 검증
   - 재현 가능한 검증 스펙 `artifacts/e2e-browser-generation.spec.ts`를 추가했습니다.
   - 실제로 `http://127.0.0.1:3010/?theme=light`에 접속해
     - 파일 업로드
     - `콘텐츠 만들기` 클릭
     - 결과 화면의 `영상 원본 열기`, `게시글 원본 열기`
     - 두 링크가 모두 `http://127.0.0.1:8000/media/projects/...`를 가리키는지
     - 각 미디어 응답이 `200`인지
     를 확인했습니다.
   - 실행 로그 기준 결과:
