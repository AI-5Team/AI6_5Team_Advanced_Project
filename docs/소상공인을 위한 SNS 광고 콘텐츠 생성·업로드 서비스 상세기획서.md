## 1. 서비스 정의

### 1-1. 서비스 한 줄 정의

**사진 몇 장과 몇 번의 선택만으로, 소상공인이 SNS용 숏폼 광고와 게시글을 만들고 업로드 또는 예약 발행까지 할 수 있게 돕는 서비스**

### 1-2. 해결하려는 문제

소상공인이 SNS 홍보를 하려면 보통 아래 순서를 거칩니다.

`사진 촬영 → 사진 보정 → 문구 작성 → 영상/이미지 편집 → 플랫폼별 규격 수정 → 업로드 → 예약 관리`

이 과정에는 다음 문제가 있습니다.

- 제작 도구와 업로드 도구가 분리되어 있습니다.
- 프롬프트 입력과 편집 툴 조작이 어렵습니다.
- 플랫폼별 형식 차이로 반복 작업이 많습니다.
- 꾸준히 올려야 하지만 시간을 계속 쓰기 어렵습니다.

### 1-3. 우리 서비스의 해결 방식

이 서비스는 범용 AI 생성기가 아니라, **소상공인 광고 운영 전용 툴**로 정의합니다.

- 사용자는 자유 입력보다 선택형 흐름으로 진입합니다.
- 결과물은 `숏폼 광고 영상`과 `게시글용 콘텐츠` 두 축으로 제한합니다.
- 생성이 끝나면 업로드 또는 예약 발행까지 이어집니다.
- 지역 키워드와 업종 맥락을 자동 반영합니다.
- 최신 트렌드를 템플릿 추천에 반영합니다.

## 2. 목표, 비목표, 성공 조건

### 2-1. 핵심 목표

1. 프롬프트 없이 첫 결과물을 3분 안에 생성할 수 있어야 합니다.
2. 사진 업로드부터 게시 직전 단계까지 하나의 흐름으로 연결되어야 합니다.
3. 카페/음식점 기준으로 실제 시연 가능한 결과물을 제공해야 합니다.
4. 5~7초 길이의 짧은 세로형 숏폼을 안정적으로 생성해야 합니다.
5. 발표 시점에는 최소 1개 채널 기준 게시 흐름이 작동해야 합니다.

### 2-2. 비목표

아래는 초기 MVP에서 의도적으로 제외합니다.

- 전문 편집기 수준의 자유 편집 기능
- 15초 이상 장편 영상 생성
- 전 업종 지원
- 모든 SNS 채널 완전 지원
- 정교한 광고 성과 분석 대시보드
- 생성형 비디오 100% 의존형 파이프라인

### 2-3. 성공 조건

- 사용자가 사진 2~3장으로 첫 콘텐츠를 생성할 수 있습니다.
- 생성된 숏폼이 업종/목적/스타일을 식별 가능하게 반영합니다.
- 생성된 캡션이 그대로 업로드 가능한 수준입니다.
- 업로드 또는 예약 발행 시연이 끊기지 않습니다.
- 사용자 테스트에서 “직접 써볼 수 있겠다”는 반응을 확보합니다.

## 3. 타깃 사용자와 페르소나

### 3-1. 핵심 타깃

- 인스타그램, 유튜브 쇼츠, 틱톡으로 홍보하고 싶은 소상공인
- 카페, 음식점, 미용실, 네일샵, 공방, 소매점 등 시각 홍보가 중요한 업종
- 편집 도구와 프롬프트 사용이 익숙하지 않은 사용자
- 1인 또는 소규모 매장 운영자
- 전국 노출보다 지역 유입이 중요한 오프라인 매장 운영자

### 3-2. 대표 페르소나 A

- 유형: 동네 카페 사장님
- 연령대: 40대
- 현재 행동: 사진은 찍지만 릴스는 거의 만들지 못합니다.
- 니즈: 신메뉴, 할인, 매장 분위기를 빠르게 홍보하고 싶습니다.
- 불편: 문구 작성, 자막 구성, 업로드 타이밍 관리가 어렵습니다.

### 3-3. 대표 페르소나 B

- 유형: 음식점 운영자
- 연령대: 30대 후반
- 현재 행동: 배달앱과 인스타그램을 병행하지만 영상은 자주 못 올립니다.
- 니즈: 오늘의 메뉴와 점심 특선을 짧고 자주 알리고 싶습니다.
- 불편: 플랫폼마다 따로 업로드하는 것이 번거롭습니다.

## 4. 핵심 가치 제안

### 4-1. 프롬프트가 아니라 선택으로 시작합니다

사용자는 아래 정보만 선택 또는 입력하면 됩니다.

- 업종
- 매장 위치
- 홍보 목적
- 스타일
- 게시 채널
- 사진

### 4-2. 템플릿형 숏폼을 빠르게 만듭니다

빈 화면에서 직접 편집하는 것이 아니라, 템플릿 기반으로 영상 구조를 자동 조합합니다.

### 4-3. 생성에서 끝나지 않고 업로드까지 연결합니다

“만들었으니 알아서 올리세요”가 아니라, 업로드 또는 예약 발행 단계까지 한 흐름으로 묶습니다.

### 4-4. AI의 어색함을 스타일로 흡수합니다

고급 광고처럼만 보이게 하는 것이 아니라, 일부 템플릿은 레트로/B급 감성을 의도적으로 채택해 어색함을 콘셉트로 전환합니다.

### 4-5. 동네 맥락이 있는 광고를 만듭니다

지역명 기반 카피와 해시태그를 자동 생성해 전국 공통형 문구보다 더 맥락 있는 결과물을 제공합니다.

## 5. MVP 범위

### 5-1. MVP 포함 기능

| 구분 | 기능 | 우선순위 | 포함 여부 |
|---|---|---|---|
| 계정 | 기본 로그인/세션 유지 | P0 | 포함 |
| 매장정보 | 업종/지역 입력 | P0 | 포함 |
| 생성 | 사진 업로드 및 가이드 | P0 | 포함 |
| 생성 | 이미지 자동 보정 | P0 | 포함 |
| 생성 | 필요 시 배경 제거 | P1 | 포함 |
| 생성 | 숏폼 5~7초 생성 | P0 | 포함 |
| 생성 | 게시글 이미지/카피 생성 | P0 | 포함 |
| 생성 | 지역 키워드 반영 | P0 | 포함 |
| 편집 | 빠른 수정 버튼 | P1 | 포함 |
| 업로드 | 1개 채널 업로드 또는 업로드 보조 | P0 | 포함 |
| 업로드 | 예약 발행 | P1 | 포함 |
| 추천 | 이번 주 추천 템플릿 | P1 | 포함 |
| 관리 | 생성 이력 조회 | P1 | 포함 |

### 5-2. MVP 제외 기능

| 기능 | 제외 이유 |
|---|---|
| 자유형 타임라인 편집기 | 일정 대비 구현량이 큼 |
| 모든 채널 동시 업로드 완전 지원 | API/권한/정책 리스크가 큼 |
| 고급 성과 대시보드 | 시연 핵심 기능이 아님 |
| 긴 영상 생성 | 비용과 처리 시간이 큼 |
| 업종 10개 이상 지원 | 템플릿/카피 규칙이 과도하게 늘어남 |

### 5-3. 발표 시연 필수 조건

- 업종 2개: 카페, 음식점
- 스타일 3개: 기본, 친근함, 하찮고 웃김
- 템플릿 3개 이상
- 세로형 숏폼 1종 이상 생성
- 게시글형 콘텐츠 1종 이상 생성
- 최소 1개 채널 업로드 흐름 시연

## 8. 화면 상세 명세

이 장은 실제 와이어프레임과 프론트 구현의 기준으로 사용합니다.

### SC-01. 로그인 화면

- 목적: 사용자를 서비스에 진입시킵니다.
- 진입 조건: 비로그인 상태
- 주요 컴포넌트
  - 이메일 입력
  - 비밀번호 입력
  - 로그인 버튼
  - 간편 로그인 버튼(선택)
- 예외 처리
  - 이메일 형식 오류
  - 비밀번호 공란
  - 로그인 실패 메시지

### SC-02. 홈 화면

- 목적: 새 콘텐츠 생성 시작점과 최근 작업 확인
- 진입 조건: 로그인 완료
- 주요 컴포넌트
  - 새 콘텐츠 만들기 버튼
  - 최근 생성 콘텐츠 카드
  - 이번 주 추천 템플릿
  - 예약 발행 예정 카드
  - SNS 연동 상태 요약
- CTA 우선순위
  1. 새 콘텐츠 만들기
  2. 최근 작업 이어서 하기

### SC-03. 생성 마법사 1단계: 업종 선택

- 목적: 카피 규칙과 템플릿 추천의 기준 업종 수집
- 입력 필드

| 필드명 | 타입 | 필수 | 값 |
|---|---|---|---|
| 업종 | 단일 선택 | 예 | 카페, 음식점 |

- 버튼
  - 다음
  - 취소
- 검증
  - 업종 미선택 시 다음 버튼 비활성화

### SC-04. 생성 마법사 2단계: 매장 위치 입력

- 목적: 지역 키워드 생성의 기준 위치 수집
- 입력 필드

| 필드명 | 타입 | 필수 | 제한 |
|---|---|---|---|
| 매장 위치 | 텍스트 | 예 | 2~20자 |
| 상세 위치 | 텍스트 | 아니오 | 0~30자 |

- placeholder 예시
  - 성수동
  - 역삼역 근처
  - 서면 1번가
- 검증
  - 매장 위치 2자 미만 불가
  - 특수문자 반복 입력 제한

### SC-05. 생성 마법사 3단계: 홍보 목적 선택

- 목적: 카피 구조와 템플릿 매칭 기준 수집
- 입력 필드

| 필드명 | 타입 | 필수 | 값 |
|---|---|---|---|
| 홍보 목적 | 단일 선택 | 예 | 신메뉴, 할인/행사, 후기 강조, 위치 홍보 |

- 검증
  - 목적 미선택 시 다음 버튼 비활성화

### SC-06. 생성 마법사 4단계: 스타일 선택

- 목적: 편집 규칙과 문구 톤 확정
- 입력 필드

| 필드명 | 타입 | 필수 | 값 |
|---|---|---|---|
| 스타일 | 단일 선택 | 예 | 기본, 친근함, 하찮고 웃김 |

- 선택 카드 설명
  - 기본: 가장 무난한 홍보형
  - 친근함: 부드럽고 접근성 높은 톤
  - 하찮고 웃김: B급/레트로 감성

### SC-07. 생성 마법사 5단계: 채널 선택

- 목적: 출력 포맷과 업로드 경로 확정
- 입력 필드

| 필드명 | 타입 | 필수 | 값 |
|---|---|---|---|
| 게시 채널 | 복수 선택 | 예 | 인스타그램, 유튜브 쇼츠, 틱톡 |

- 검증
  - 최소 1개 채널 선택

- 이벤트성 해시태그는 목적에 따라 추가

## 12. 업로드/예약 발행 정책

### 12-1. 채널 지원 정책

| 채널 | MVP 상태 | 비고 |
|---|---|---|
| 인스타그램 릴스/피드 | 우선 검토 | 실제 API/권한 범위 확인 필요 |
| 유튜브 쇼츠 | 우선 검토 | 업로드 흐름 검증 필요 |
| 틱톡 | 확장 검토 | 정책 및 권한 확인 필요 |

정확한 자동 업로드 지원 범위는 구현 전에 각 플랫폼 정책 검증 후 확정합니다.

### 12-2. 업로드 방식

- `자동 업로드`: 연동된 계정으로 직접 게시
- `업로드 보조`: 파일과 캡션을 정리해 사용자가 수동 업로드하도록 지원

### 12-3. 게시 실패 분류

| 실패 유형 | 설명 | 사용자 안내 |
|---|---|---|
| AUTH_ERROR | 권한 만료/계정 미연동 | 계정 재연동 요청 |
| FORMAT_ERROR | 파일 형식 부적합 | 포맷 재생성 안내 |
| NETWORK_ERROR | 네트워크/외부 API 문제 | 잠시 후 재시도 |
| POLICY_ERROR | 플랫폼 정책 위반 가능성 | 문구/포맷 확인 요청 |

### 12-4. 예약 발행 운영 규칙

- 예약은 현재 시각 이후만 허용합니다.
- 발행 실패 시 최소 1회 재시도합니다.
- 재시도 실패 시 상태를 `failed`로 전환하고 알림합니다.

## 15. 예외 처리 및 검증 규칙

### 15-1. 입력 검증

- 업종 미선택 시 생성 진행 불가
- 지역 입력 2자 미만 불가
- 채널 1개 이상 선택 필수
- 사진 최소 1장 필수
- 지원하지 않는 파일 형식 업로드 불가

### 15-2. 생성 실패 예외

- 사진 전처리 실패
- 카피 생성 실패
- 영상 렌더링 실패
- 저장 실패

각 실패는 공통 오류로 묶지 않고, 실패 단계별로 분리 저장합니다.

### 15-3. 게시 실패 예외

- 계정 미연동
- 권한 만료
- 외부 플랫폼 응답 실패
- 예약 시간 오류

### 15-4. 사용자 메시지 원칙

- 기술 내부 오류 문구를 그대로 노출하지 않습니다.
- 사용자가 다음 행동을 알 수 있게 메시지를 씁니다.
- 예시: “인스타그램 계정 연결이 만료되었습니다. 다시 연결해 주세요.”

## 17. 기술 구현 방향

### 17-1. 생성 파이프라인 방향

기본 전략은 `템플릿 기반 영상 조합 + 자동 카피 생성 + 채널별 후처리`입니다.

### 17-2. 모듈 구분

1. 프론트엔드: 생성 마법사, 결과 확인, 이력, 연동 상태
2. 백엔드 API: 프로젝트 생성, 상태 조회, 업로드 관리
3. 전처리 모듈: 이미지 보정, 배경 제거
4. 렌더링 모듈: 템플릿 기반 숏폼 생성
5. 카피 생성 모듈: 목적/업종/지역 기반 문구 생성
6. 업로드 워커: 즉시 업로드/예약 발행 처리
7. 트렌드 수집기: 추천 템플릿 갱신

### 17-3. 생성형 비디오 실험 위치

생성형 비디오는 메인 파이프라인이 아니라 아래 중 일부를 대체하는 **실험 모듈**로 둡니다.

- 인트로 모션
- 배경 모션
- 전환 효과 일부

즉, 기본 MVP는 생성형 비디오가 없어도 완성되어야 합니다.

### 17-4. 권장 기술 스택안

현재 저장소에 기술 스택이 확정되어 있지 않으므로, 팀이 공통 목표로 개발을 시작할 수 있도록 아래를 **권장안**으로 제시합니다.

| 영역 | 권장안 | 선택 이유 |
|---|---|---|
| 프론트엔드 | Next.js + TypeScript | 화면 흐름, 상태 관리, 배포 편의성 |
| 백엔드 API | FastAPI | AI/비동기 처리와 Python 생태계 연계 용이 |
| 워커 | Python worker + Redis queue | 전처리/렌더링/업로드 작업 분리 |
| DB | PostgreSQL | 상태 관리와 이력 저장 안정성 |
| 캐시/큐 | Redis | 비동기 작업 큐와 상태 캐싱 |
| 파일 저장소 | GCS 또는 S3 호환 스토리지 | 이미지/영상 결과물 저장 |
| 인증 | 자체 세션 또는 JWT | MVP 기준 단순 인증 우선 |
| 배포 | Web/API/Worker 분리 배포 | 장애 영향 분리 |

이 권장안의 핵심은 “프론트는 TS, AI/렌더링은 Python”으로 역할을 분리해 팀원 간 구현 경계를 명확히 하는 데 있습니다.

### 17-5. 시스템 아키텍처 초안

```text
[Web Frontend]
  - 생성 마법사
  - 결과 화면
  - 이력/예약/연동 UI
        |
        v
[API Server]
  - 입력 검증
  - 프로젝트 생성
  - 상태 조회
  - 업로드 요청 등록
        |
        +----------------------+
        |                      |
        v                      v
[PostgreSQL]              [Redis Queue]
  - 프로젝트                - 전처리 작업
  - 결과 메타데이터          - 렌더링 작업
  - 업로드 상태              - 업로드 작업
        |                      |
        |                      v
        |                 [Worker]
        |                   - 이미지 전처리
        |                   - 카피 생성
        |                   - 숏폼 렌더링
        |                   - 업로드/예약 발행
        |                      |
        +-----------> [Object Storage]
                           - 원본 이미지
                           - 전처리 이미지
                           - 결과 영상/이미지
```

### 17-6. 모듈 책임 분리 원칙

공통 목표로 개발하려면 “어디까지가 누구 책임인지”가 먼저 고정되어야 합니다.

| 모듈 | 책임 | 하지 않는 일 |
|---|---|---|
| Web | 사용자 입력 수집, 결과 표시, 상태 폴링, 빠른 수정 UI | 영상 렌더링, 외부 업로드 직접 처리 |
| API | 입력 검증, 프로젝트 생성, 작업 큐 등록, 상태 응답 | 무거운 렌더링 처리 |
| Worker | 전처리, 카피 생성, 렌더링, 업로드, 예약 작업 실행 | 사용자 세션 관리, 화면 렌더링 |
| DB | 프로젝트/상태/이력 저장 | 대용량 원본 파일 저장 |
| Object Storage | 이미지/영상 파일 저장 | 상태 관리 |

### 18-9. 비동기 처리 규칙

- MVP에서는 WebSocket보다 `상태 조회 polling`을 우선합니다.
- 프론트는 생성 중 2~3초 간격으로 상태를 조회합니다.
- 업로드/예약 발행 상태는 5초 간격 조회를 기본으로 합니다.
- 장기적으로는 SSE 또는 WebSocket으로 변경 가능하지만, MVP 공통 목표는 polling 기준으로 맞춥니다.

### 18-10. 외부 연동 경계

외부 채널 연동은 Worker에서 직접 호출하되, API와의 계약은 아래처럼 단순화합니다.

| 내부 입력 | 외부 변환 책임 |
|---|---|
| projectId | Worker |
| channel | Worker |
| caption | Worker |
| hashtags | Worker |
| media URL | Worker |

즉, 프론트는 외부 플랫폼별 복잡한 정책을 몰라도 되고, API/Worker 계층에서 흡수합니다.

## 19. 역할 분배안

### 19-1. 운영 원칙

- 메인 기능인 `숏폼 제작`은 전원 공동 개발합니다.
- 각 팀원은 개별 방식으로 숏폼 프로토타입을 구현합니다.
- 이후 장점을 비교해 통합 버전을 만듭니다.
- 보조 기능은 개인 주담당을 나누어 병행 개발합니다.
- 실험 과정과 결과는 각 담당자가 반드시 문서화합니다.

### 19-2. 팀원별 담당

| 이름 | 공동 파트 | 개인 주담당 | 상세 책임 |
|---|---|---|---|
| 임창현 | 숏폼 제작 공동 개발 | PM/기획/통합 | 상세기획서 관리, 공통 요구사항 정리, 결과 통합, 발표 시연 구조 정리 |
| 신유철 | 숏폼 제작 공동 개발 | 업로드/예약 발행 | SNS 계정 연동, 게시 흐름, 예약 발행, 업로드 실패 처리 |
| 이진석 | 숏폼 제작 공동 개발 | 카피 생성/지역 문구 | 목적별 문구 규칙, 지역 키워드 반영, 해시태그 생성 |
| 최무영 | 숏폼 제작 공동 개발 | 이미지 전처리 | 사진 품질 보정, 배경 제거, 촬영 가이드 설계 |
| 서유종 | 숏폼 제작 공동 개발 | 트렌드 추천 | 트렌드 수집, 추천 템플릿 로직, 추천 데이터 정리 |

### 19-3. 숏폼 공동 개발 산출물

모든 팀원은 아래 산출물을 각자 제출합니다.

1. 개인 숏폼 프로토타입
2. 사용한 템플릿 규칙 설명
3. 생성 속도 및 품질 메모
4. 장점/한계 정리
5. 스크린샷 또는 영상 결과

### 19-4. 통합 회의 기준

통합 시 아래 항목으로 비교합니다.

- 시연 안정성
- 구현 복잡도
- 결과물 일관성
- 템플릿 확장성
- 속도

## 22. 문서화 운영 규칙

사용자께서 주신 팀 규칙에 따라, 연구나 실험은 과정과 결과 문서화를 필수로 합니다.

### 22-1. 필수 문서 종류

| 문서 종류 | 목적 | 작성 책임 |
|---|---|---|
| 상세기획서 | 기준 문서 | 임창현 |
| 실험 로그 | 과정/결과 기록 | 각 담당자 |
| 기술 의사결정 문서 | 선택 이유 기록 | 해당 의사결정 담당자 |
| 실패 사례 문서 | 오류와 대응 기록 | 해당 담당자 |
| Daily 협업일지 | 일일 진행 기록 | 전원 |
| 사용자 테스트 기록 | 피드백 누적 | 테스트 담당자 |

### 22-2. 권장 폴더 구조

문서 폴더는 최소 아래 구조로 운영하는 것을 권장합니다.

```text
docs/
├─ planning/
│  ├─ 소상공인을 위한 SNS 광고 콘텐츠 생성·업로드 서비스 상세기획서.md
│  ├─ IA.md
│  └─ screen-flow.md
├─ experiments/
│  ├─ EXP-01-숏폼-비교.md
│  ├─ EXP-02-스타일-비교.md
│  └─ EXP-03-전처리-비교.md
├─ adr/
│  ├─ ADR-001-템플릿-기반-숏폼-채택.md
│  ├─ ADR-002-업로드-보조-모드-도입.md
│  └─ ADR-003-생성형-비디오-실험-분리.md
├─ daily/

   ├─ test-scenario-01.md
   ├─ user-feedback-01.md

```

### 22-3. 문서 파일명 규칙

- 실험 로그: `EXP-번호-주제.md`
- ADR: `ADR-번호-결정사항.md`
- 일지: `YYYY-MM-DD-이름.md`
- 테스트 기록: `test-scenario-번호.md`, `user-feedback-번호.md`

### 22-4. 기록 원칙

1. 실험한 날 바로 기록합니다.
2. 실패한 결과도 기록합니다.
3. 스크린샷/영상 링크를 남깁니다.
4. 왜 그렇게 판단했는지 적습니다.
5. 발표 직전 몰아서 정리하지 않습니다.

## 23. 테스트 및 수용 기준

### 23-1. 사용자 테스트 시나리오

1. 카페 사장님 시나리오로 신메뉴 홍보 생성
2. 음식점 시나리오로 점심 특선 홍보 생성
3. 결과 수정 후 업로드 또는 예약 발행 수행

### 23-2. 테스트 관찰 포인트

- 첫 생성까지 걸리는 시간
- 어디에서 멈추는지
- 어떤 용어를 어려워하는지
- 업로드 흐름을 믿는지
- 결과물이 실제로 쓸 만하다고 느끼는지

### 23-3. 수용 기준

| 항목 | 기준 |
|---|---|
| 첫 결과물 생성 | 3분 이내 |
| 주요 플로우 완수율 | 80% 이상 목표 |
| 업로드/업로드 보조 성공 | 시연 기준 1개 채널 이상 |
| 숏폼 결과 식별성 | 업종/목적/스타일 식별 가능 |
| 문서화 누락 | 없음 |

## 24. 리스크와 대응 방안

### 24-1. 플랫폼 정책 리스크

- 위험: 자동 업로드 지원이 제한될 수 있습니다.
- 대응: 업로드 보조 모드를 fallback으로 둡니다.

### 24-2. 생성 품질 리스크

- 위험: 결과가 어색할 수 있습니다.
- 대응: 자유 생성보다 템플릿 중심 전략을 유지합니다.

### 24-3. 일정 리스크

- 위험: 전원이 공동 개발하는 숏폼 파트 통합이 늦어질 수 있습니다.
- 대응: 비교 기준과 마감 시점을 초기에 고정합니다.

### 24-4. 문서화 누락 리스크

- 위험: 실험은 했지만 기록이 남지 않을 수 있습니다.
- 대응: Daily 협업일지 + 실험 로그를 병행 관리합니다.
